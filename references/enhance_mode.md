# Enhance Mode Reference / エンハンスモード リファレンス

既存プロダクトの改善・機能追加の要件定義に特化したモード。
インターネットリサーチとユーザーヒアリングを同時進行し、変更要件を構造化する。

---

## 1. Phase 1: 対話型リサーチの進め方

エンハンスモードでは Phase 1 でリサーチとヒアリングを並行実行する。

### ステップ

1. **基本情報の収集**: プロダクト名、URL（あれば）、概要を聞く
2. **並行リサーチ**: WebSearch でプロダクト情報を検索、WebFetch で公式ページを取得
3. **信頼度分類**: 収集した情報を分類してユーザーに提示（下記参照）
4. **ユーザー確認**: 提示情報の正誤を確認、不足情報を直接質問
5. **追加資料の受付**: ドキュメント、スクリーンショット、内部仕様等のファイル提供を受け付ける
6. **現状サマリー記録**: 確認済み情報を `designs/README.md` の「現状サマリー」セクションに記録
7. **変更背景のヒアリング**: なぜ変更が必要か、通常のプロジェクト憲章セクションに反映

### 情報の信頼度分類

| 分類     | マーク | 基準                                       | 扱い                       |
| -------- | ------ | ------------------------------------------ | -------------------------- |
| 確認済み | ✅     | 公式ソースから取得、またはユーザー確認済み | そのまま記録               |
| 要確認   | ⚠️     | 第三者ソース、または古い情報の可能性あり   | ユーザーに確認してから記録 |
| 不明     | ❓     | 情報が見つからない                         | ユーザーに直接質問         |

### 安全ルール（CRITICAL）

- **公式ソース優先**: 公式サイト・公式ドキュメントを最優先で参照する
- **クロスチェック**: 非公式情報は2ソース以上で確認してから提示
- **推測禁止**: 推測を事実として記録しない。確信がない情報は必ず「⚠️ 要確認」として提示
- **ソースURL必須**: 全リサーチ結果にソースURLを付記する
- **フォールバック**: WebSearch/WebFetch が利用不可の場合、ユーザーに手動で情報提供を依頼する

### Phase 1 質問カタログ（エンハンスモード用）

**基本情報:**

1. プロダクトの名前とURLを教えてください
2. このプロダクトの主な機能を簡単に説明してください
3. ターゲットユーザーは誰ですか？
4. 技術スタックは何を使っていますか？（わかる範囲で）

**変更の背景:** 5. 今回の変更のきっかけは何ですか？（ユーザーフィードバック、事業戦略、技術負債等）6. この変更で達成したい目標は何ですか？7. 変更のスコープ: 影響範囲はプロダクト全体？特定の機能？

### 現状サマリーセクション（README.md に追加）

```markdown
## 10. Current State Summary / 現状サマリー

- **Product Name**: [プロダクト名]
- **Product URL**: [URL]
- **Overview**: [プロダクト概要]
- **Key Features**: [主要機能一覧]
- **Tech Stack**: [技術スタック]
- **Target Users**: [ターゲットユーザー]
- **Sources**: [情報ソースURL一覧]
```

## 2. Phase 2: 変更ヒアリング

エンハンスモードの Phase 2 は「変更ヒアリング」形式で進行する。

### Round 1: 変更の特定 (3-5問)

1. **追加したい機能・振る舞いは何ですか？**
   What features or behaviors do you want to add?

2. **修正が必要な既存機能はありますか？どう変えたいですか？**
   Are there existing features that need modification? How?

3. **削除・廃止する機能はありますか？**
   Are any features being removed or deprecated?

4. **この変更のきっかけは？**（ユーザーフィードバック、事業要求、技術負債）
   What triggered this change?

5. **変更しない部分は何ですか？**（スコープのアンカリング）
   What parts of the product will NOT change?

### Round 2: 変更の詳細 (3-5問)

6. **各変更について: 現状の振る舞いと目標の振る舞いを教えてください**
   For each change: describe current behavior and target behavior.

7. **変更による他機能への影響はありますか？**
   Will these changes affect other features?

8. **データモデルの変更は必要ですか？**
   Are data model changes required?

9. **既存ユーザーへの影響は？マイグレーションは必要ですか？**
   Impact on existing users? Is migration needed?

10. **リリースの優先順位は？段階的リリースですか？**
    Release priority? Phased rollout?

### Round 3 (任意): 影響分析 (3-5問)

11. **既存機能との衝突・依存関係はありますか？**
12. **リグレッションリスクが高い箇所はどこですか？**
13. **パフォーマンスへの影響はありますか？**
14. **セキュリティ上の考慮事項はありますか？**
15. **この変更に伴うドキュメント更新は必要ですか？**

## 3. デルタFRフォーマット

エンハンスモードでは、通常のFR項目に加えて以下のオプションフィールドを使用する:

```markdown
#### FR-XXX: [要件タイトル]

- **Change Type**: Add / Modify / Remove
- **説明 / Description**: [変更内容]
- **Current Behavior**: [（Modify時）現在の振る舞い]
- **Target Behavior**: [（Modify時）目標の振る舞い]
- **アクター / Actor**: [操作するアクター]
- **主フロー / Main Flow**: [変更後の主フロー]
- **例外フロー / Exception Flow**: [変更後の例外フロー]
- **影響範囲 / Impact Scope**: [この変更が影響する既存機能]
- **優先度 / Priority**: Must / Should / Could
```

**Change Type の使い分け:**

- **Add**: 完全に新しい機能の追加。Current Behavior は不要
- **Modify**: 既存機能の変更。Current Behavior と Target Behavior の両方を記載
- **Remove**: 機能の削除・廃止。移行計画を例外フローに記載

## 4. 品質スコアリング調整

エンハンスモードは Full Mode と同じ5次元100pt評価（合格70pt）だが、チェック基準をデルタ要件向けに調整する。

| 次元         | エンハンスモードでの評価ポイント                                           |
| ------------ | -------------------------------------------------------------------------- |
| 網羅性       | 全変更要求がFR化されているか。既存機能への影響が特定されているか           |
| 具体性       | Current Behavior と Target Behavior が明確に区別されているか               |
| テスト可能性 | リグレッションテスト基準を含むか（新規テストだけでなく既存テストへの影響） |
| 一貫性       | 新要件が既存プロダクトの振る舞い・用語と矛盾しないか                       |
| 追跡可能性   | 変更要求 → デルタFR → デルタUS の紐付けが完全か                            |

## 5. モード設定値

| パラメータ       | Enhance                          |
| ---------------- | -------------------------------- |
| 対象             | 既存プロダクトの改善             |
| 所要時間         | 30-45分                          |
| Phase 1 ラウンド | 1-2（リサーチ並行）              |
| Phase 2 ラウンド | 2-3（変更ヒアリング）            |
| FR項目           | 全10項目（デルタフィールド追加） |
| 品質次元         | 5 x 20pt = 100pt                 |
| 合格ライン       | 70/100 (70%)                     |
| 自動スキップ     | Phase 5                          |
| スキップ可能     | Phase 3, 4C（ユーザー選択）      |

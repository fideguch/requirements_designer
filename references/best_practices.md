## 良い要件 vs 悪い要件

### 機能要件 (FR)

| ❌ 悪い例                  | ✅ 良い例                                                                                                      | 改善ポイント                       |
| -------------------------- | -------------------------------------------------------------------------------------------------------------- | ---------------------------------- |
| ユーザーはログインできる   | ユーザーはメールアドレスとパスワードでログインできる。パスワードは8文字以上、英数混在必須                      | 具体的な入力形式と制約を明記       |
| 検索が速い                 | 検索結果は1秒以内に表示される（結果100件以下の場合）                                                           | 定量的な基準と条件を明記           |
| エラー時に適切に対応する   | 決済失敗時はエラーコードと「お支払いが完了しませんでした。再度お試しください」を表示し、カートの内容は保持する | 具体的なエラー処理を明記           |
| 管理者はデータを管理できる | 管理者はCSV形式で注文データを日付範囲指定でエクスポートできる。出力項目: 注文ID, 日時, 金額, ステータス        | CRUD操作を具体的に分解             |
| 通知を送る                 | 注文確定から5分以内にメールで注文確認通知を送信する。件名: 「ご注文を承りました（注文番号: {order_id}）」      | 通知の条件・タイミング・内容を明記 |

### 非機能要件 (NFR)

| ❌ 悪い例            | ✅ 良い例                                                                                     | 改善ポイント               |
| -------------------- | --------------------------------------------------------------------------------------------- | -------------------------- |
| システムは安全である | 通信はTLS 1.3で暗号化。パスワードはbcrypt（コスト12）でハッシュ化。セッションは30分で失効     | 具体的な技術と数値を明記   |
| 高い可用性を確保する | 月間稼働率99.9%（年間ダウンタイム8.76時間以内）。計画メンテナンスは月1回、深夜2:00-4:00に実施 | 数値目標と例外条件を明記   |
| スケーラブルである   | 同時接続1,000ユーザーまで対応。負荷超過時はキューイングで処理し、エラーにはしない             | 上限値と超過時の挙動を明記 |
| 使いやすい           | 初回ユーザーが注文完了まで3タップ以内。ユーザビリティテストでタスク完了率90%以上              | 定量的な測定基準を明記     |

## 要件定義でよくある落とし穴

### 1. スコープクリープの兆候

以下のフレーズが出てきたら注意:

- 「ついでに○○も」
- 「将来的には△△も考えると」
- 「どうせ作るなら□□も」

→ Out of Scope に明記して、将来のフェーズに回す。

### 2. 暗黙の前提

チーム内で「当然」と思われていて文書化されない要件:

- 認証・認可の仕組み
- エラーハンドリングの方針
- ログ出力の方針
- データバックアップ
- 多言語対応の有無

→ Phase 3 で意図的に確認する。

### 3. ステークホルダーの不一致

異なるステークホルダーが同じ機能について異なる期待を持つ:

- 営業「全ての機能を初回リリースに」
- 開発「技術的に3ヶ月では無理」
- 経営「予算内でMVPを」

→ 優先度（MoSCoW）で合意を取り、README.mdに記録する。

## NFRカテゴリ別のよくある見落とし

### パフォーマンス

- 検索機能のレスポンスタイム（データ量増加時）
- ファイルアップロード/ダウンロードの上限サイズ
- バッチ処理の実行時間制限

### 可用性

- 計画メンテナンス時の挙動（メンテナンスページ？）
- 障害時のフォールバック戦略
- 冗長構成の必要性

### セキュリティ

- セッション管理（タイムアウト、同時ログイン制限）
- 個人情報の暗号化（保存時・通信時）
- 脆弱性スキャンの頻度
- CORS設定
- CSRFトークン

### スケーラビリティ

- 1年後のデータ量予測
- 水平スケーリング vs 垂直スケーリング
- キャッシュ戦略

### ユーザビリティ

- モバイルファーストかデスクトップファーストか
- アクセシビリティ（WCAG 2.1 AA）
- ローディング状態の表示
- エラーメッセージのトーン

### 保守性

- テストカバレッジ目標
- コーディング規約
- ドキュメント更新ルール
- 技術的負債の管理方針

### 互換性

- ブラウザ対応範囲
- API後方互換性
- モバイルOS最低バージョン

### 法規制

- 個人情報保護法（日本）
- GDPR（EU向けの場合）
- PCI DSS（決済情報の場合）
- 特定商取引法（EC）

### データ

- RPO / RTO の定義
- データ移行計画（既存システムからの移行）
- データ削除ポリシー（退会時等）

### 運用

- ログの保持期間
- 監視ダッシュボードの要件
- インシデント対応フロー
- デプロイ手順・ロールバック手順

## 要件の書き方テンプレート

### SMART基準

- **S**pecific（具体的）: 何を、誰が、どのように
- **M**easurable（測定可能）: 数値で測れる
- **A**chievable（達成可能）: 技術的・予算的に実現可能
- **R**elevant（関連性）: プロジェクト目標に紐づく
- **T**ime-bound（期限付き）: いつまでに必要か

### RFC 2119 のキーワード

要件の強度を明確にする:

- **MUST** / **必須**: これがないとリリースできない
- **SHOULD** / **推奨**: 原則として実装するが、正当な理由があれば省略可
- **MAY** / **任意**: あると良いが、なくても問題ない

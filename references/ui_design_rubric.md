# UI Design Quality Rubric / UIデザイン品質評価ルブリック

UIデザインの品質を5次元 x 20点 = 100点満点で評価する。

---

## 1. Design System完全性 / Design System Completeness (20点)

デザインシステムがFigma変数・スタイルとして正しく定義されているか。

### チェックポイント (各4点)

| #   | チェック項目                                                                        | 確認方法                        |
| --- | ----------------------------------------------------------------------------------- | ------------------------------- |
| 1   | カラー変数が全ロール分（Primary/Secondary/Accent/Neutral/Semantic）定義されているか | Figma変数コレクションを確認     |
| 2   | タイポグラフィスタイルが全レベル（H1-H3/Body/Caption/Overline）定義されているか     | Figmaテキストスタイル一覧を確認 |
| 3   | スペーシング変数（4px基準のスケール）が定義されているか                             | Figma変数コレクションを確認     |
| 4   | 角丸・シャドウの変数/スタイルが定義されているか                                     | Figmaエフェクトスタイル確認     |
| 5   | デザイントークンがDesign Systemページで視覚的に文書化されているか                   | ドキュメントフレームの有無      |

### スコアガイド

- **18-20**: 全トークンがFigma変数として定義、ドキュメントフレームあり
- **14-17**: 主要トークンは定義済みだが1-2カテゴリが不足
- **10-13**: カラーとタイポのみ定義、スペーシング/角丸/シャドウが未定義
- **6-9**: カラーのみ定義、他は直値で適用
- **0-5**: デザインシステムがほぼ未整備

---

## 2. 画面カバレッジ / Screen Coverage (20点)

要件から導出された画面がどの程度カバーされているか。

### チェックポイント (各4点)

| #   | チェック項目                                             | 確認方法                                                |
| --- | -------------------------------------------------------- | ------------------------------------------------------- |
| 1   | 全USが画面インベントリにマッピングされているか           | ui_design_brief.mdの画面インベントリ vs user_stories.md |
| 2   | Must優先度の全USにワイヤーフレームが存在するか           | Figma Wireframesページを確認                            |
| 3   | Must優先度の全USにモックアップが存在するか               | Figma Mockupsページを確認                               |
| 4   | Should優先度のUSの50%以上にワイヤーフレームが存在するか  | 画面インベントリのStatus列                              |
| 5   | 共通ナビゲーション・ヘッダー・フッターが定義されているか | Design Systemページのコンポーネント                     |

### スコアガイド

- **18-20**: 全Must USにWF+MK、全Should USにWF、共通コンポーネント定義済み
- **14-17**: 全Must USにWF+MK、Should USのカバーが一部不足
- **10-13**: Must USの大半にWFあるがMKが不足
- **6-9**: Must USの一部のみWF
- **0-5**: 画面カバレッジが断片的

---

## 3. ビジュアルヒエラルキー / Visual Hierarchy (20点)

各画面の視覚的階層が明確で、ユーザーが迷わず操作できるか。

### チェックポイント (各4点)

| #   | チェック項目                                                | 確認方法                                |
| --- | ----------------------------------------------------------- | --------------------------------------- |
| 1   | 各画面にヒーロー要素（最も目立つ要素）が1つ明確に存在するか | Squintテスト（画面を45%に縮小して確認） |
| 2   | CTAボタンの視認性は十分か（コントラスト、サイズ、配置）     | CTAが3秒以内に見つかるか                |
| 3   | 情報の優先度がレイアウトの上→下、左→右の順に並んでいるか    | Z字/F字パターンとの整合性               |
| 4   | ナビゲーション構造が全画面で一貫しているか                  | 画面間のヘッダー・サイドバー比較        |
| 5   | テキストの見出し階層（H1→H2→H3→Body）が正しく使われているか | フォントサイズ・ウェイトの段階確認      |

### スコアガイド

- **18-20**: 全画面でヒーロー要素が明確、CTA目立つ、ナビ一貫、階層正確
- **14-17**: 大半の画面は良好だが1-2画面でヒーロー不明確
- **10-13**: ヒーロー要素はあるがCTAの視認性やナビ一貫性に課題
- **6-9**: 視覚的階層が不明確、要素が均等に並んでいる
- **0-5**: 階層がほぼなく、ユーザーがどこを見るべきか不明

### ui-ux-pro-max チェック項目

- [ ] 各セクションで1つのヒーロー要素が120点、残りが60点（70点の均一を避ける）
- [ ] セクション間隔 ≥ 112px (py-28)
- [ ] ボタンは白背景/黒テキスト（Primary）、白ボーダー（Secondary）
- [ ] 背景は単色のみ（グラデーション・ノイズテクスチャなし）
- [ ] アイコンはLucide等（絵文字不使用）

## 4. アクセシビリティ / Accessibility (20点)

WCAG 2.1基準に準拠しているか。

### チェックポイント (各4点)

| #   | チェック項目                                                                         | 確認方法                       |
| --- | ------------------------------------------------------------------------------------ | ------------------------------ |
| 1   | テキストのコントラスト比が基準を満たしているか（通常テキスト≥4.5:1、大テキスト≥3:1） | カラー変数のペアリングを検証   |
| 2   | タッチターゲット/クリッカブル要素が≥44px x 44pxか                                    | ボタン・リンクの最小サイズ確認 |
| 3   | フォーカス状態が全インタラクティブ要素に定義されているか                             | フォーカスリングのスタイル有無 |
| 4   | フォームラベル・エラーメッセージが視覚的に明確か                                     | ラベルの配置、エラー色の使用   |
| 5   | 色だけに依存していないか（形状・テキスト・アイコンでも情報伝達）                     | 色覚シミュレーション           |

### スコアガイド

- **18-20**: 全チェック項目クリア、Lighthouse Accessibility 100%相当
- **14-17**: コントラスト・タッチターゲットOKだがフォーカス状態が一部不足
- **10-13**: 基本的なコントラストはOKだがタッチターゲット不足、フォーカス未定義
- **6-9**: コントラスト違反が複数箇所
- **0-5**: アクセシビリティがほぼ考慮されていない

---

## 5. 要件トレーサビリティ / Requirements Traceability (20点)

デザインが要件定義ドキュメント（FR/NFR/US）と正しく紐付いているか。

### チェックポイント (各4点)

| #   | チェック項目                                           | 確認方法                                 |
| --- | ------------------------------------------------------ | ---------------------------------------- |
| 1   | 全画面が画面インベントリ（SCR-XXX）に登録されているか  | ui_design_brief.mdの画面インベントリ     |
| 2   | 各画面がソースUS（US-XXX）と紐付いているか             | 画面インベントリのSource US列            |
| 3   | USの受け入れ基準がUIに反映されているか                 | Given-When-Then → 画面遷移・状態変化     |
| 4   | NFRのユーザビリティ要件がデザインに反映されているか    | アクセシビリティ目標、レスポンシブ要件等 |
| 5   | FRの代替フロー・例外フローに対応するUI状態が存在するか | エラー画面、空状態、ローディング状態     |

### スコアガイド

- **18-20**: 完全なトレーサビリティ、全画面→US→FR紐付け、NFR反映、状態網羅
- **14-17**: 大半は紐付け済みだが、エラー/空状態が2-3画面不足
- **10-13**: 画面→US紐付けはあるが、NFR反映やフロー状態が不完全
- **6-9**: 紐付けが断片的、多くの画面でソースUSが不明
- **0-5**: トレーサビリティがほぼない

---

## スコア改善の優先順位

スコアが低い次元から優先的に改善する。具体的には:

1. **DS完全性が低い** → 不足しているトークン/変数をuse_figmaで追加定義
2. **画面カバレッジが低い** → 未作成のWF/MKをバッチ追加
3. **ビジュアルヒエラルキーが低い** → ui-ux-pro-maxルールに基づき要素の強弱を調整
4. **アクセシビリティが低い** → コントラスト比の修正、タッチターゲット拡大、フォーカス状態追加
5. **トレーサビリティが低い** → 画面インベントリの更新、エラー/空/ローディング状態の追加

改善後は必ず再スコアリングし、**デルタ（変化量）** を表示する。

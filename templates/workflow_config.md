## Mode / 実行モード

| パラメータ       | Full (デフォルト) | Light (MVP/PoC)  | Enhance (既存改善)          |
| ---------------- | ----------------- | ---------------- | --------------------------- |
| 対象             | 本番プロジェクト  | MVP/PoC/初期検証 | 既存プロダクトの改善        |
| 所要時間         | 40-60分           | 15-20分          | 30-45分                     |
| Phase 1 ラウンド | 1-2               | 1                | 1-2 (リサーチ並行)          |
| Phase 2 ラウンド | 3-5               | 2                | 2-3 (変更ヒアリング)        |
| FR項目           | 全10項目          | 必須5項目        | 全10項目 (デルタフィールド) |
| 品質次元         | 5 x 20pt = 100pt  | 3 x 20pt = 60pt  | 5 x 20pt = 100pt            |
| 合格ライン       | 70/100 (70%)      | 42/60 (70%)      | 70/100 (70%)                |
| 自動スキップ     | なし              | Phase 3, 4C, 5   | Phase 5                     |

### ライトモード FR 必須5項目

説明・アクター・主フロー・例外フロー・優先度

### ライトモード品質3次元

網羅性 (Completeness)・具体性 (Specificity)・テスト可能性 (Testability)

### エンハンスモード設定

- Phase 1 でインターネットリサーチとヒアリングを同時進行
- FR に Change Type (Add/Modify/Remove) と Current Behavior フィールドを追加可能
- Phase 5（UIデザイン）を自動スキップ（既存プロダクトは既存デザインを使用）
- 品質評価はデルタ要件向けに調整（詳細は `references/enhance_mode.md`）

## Skippable Phases Guide / スキップ可能フェーズガイド

### Phase 3: 非機能要件の抽出

**スキップが適切なケース:**

- MVP/PoC段階で、性能・セキュリティの詳細要件が不要
- 既に非機能要件が別ドキュメントで定義済み
- プロトタイプ・概念実証が目的

### Phase 4C: ユビキタス言語定義

**スキップが適切なケース:**

- 技術者のみが使う内部ツール（ユーザー向けUIが存在しない）
- 既存プロジェクトで用語が確立済み
- 小規模プロジェクトで用語の統一が自明

### Phase 5: UIデザイン

**スキップが適切なケース:**

- APIのみ / バックエンドのみのプロジェクト
- CLIツール
- UIデザインが別チーム・別ツールで実施される
- Figma MCPが利用不可
- エンハンスモード（既存プロダクトのデザインを使用するため自動スキップ）
